## 1. 开发流程 **[硬]**

> 本章全部为硬纪律。子节 1.1 给出总循环，1.2–1.5 是循环里每个 Plan 的具体规范。

### 1.1 三层循环

任何非平凡改动都走这套循环，不得跳级直接写代码：

```
项目级（一次性） brainstorm → 整体设计文档（spec + 关键 ADR + Sprint 分期/时间建议）
   └─ 每个 Sprint ↻   Sprint 规划文档（摘要） → 拆成多个有序 plan
        └─ 每个 Plan ↻   ④a spec/design + ★DoR 就绪门 → ④b writing-plans → ⑤ execute → ⑥ 隔离审查 → ⑦ 合并
```

1. **计划先行**：多步任务先出实现计划（brainstorming → spec/design + DoR → writing-plans → execute）再写代码；大改动设 review 检查点。
2. **一个 plan 一闭环**：每个 plan 必须能独立产出"可工作、可测试"的软件；跨独立子系统的需求拆成多个 plan。
3. **详细设计 just-in-time**：sprint 文档只给**摘要**；每个 plan 的详细 spec/design 在**进该 plan 时**产出（事实/探针/原型那刻才存在,前置进 sprint 文档=瀑布+漂移）。
4. **合并后回标**：plan 完成合并后，回填 spec / 计划里的 checkbox 与状态。

### 1.2 writing-plans 规范

一份合格的计划**必须**满足下列全部（顺序即流程：先过 DoR 就绪门、再拆分、每步带测试与验收）：

1. **探查优先（写计划之前）**：凡涉及**未知外部依赖**（第三方 API、外部服务、不熟悉的库），第一步先起真服务、跑通最小链路，把**真实端点 / 字段 / 响应 / 坑**记成事实文档，后续实现**以实测为准**。**禁止把猜测的 API 写进计划。**
2. **DoR 就绪门 + spec/design 拆分（进 tasks 之前,见 `superpowers:writing-specs`）**：写步骤前先产**独立的 spec.md(需求)+ design.md(设计)**,过**一道 DoR 就绪门**——8 项每项三态之一(**已决定 / 显式推迟+理由 / 待探查+探针+决策规则**),**无静默 TBD**。8 项 = ①范围出口 ②接口契约(对外契约冻结前过"第一个消费者":UI→低保真原型 / 程序→SDK·样例;纯内部豁免)③数据模型 ④外部依赖探查事实(=本节 1,实测成文)⑤行为·边界·并发·威胁 ⑥NFR(安全/隔离/性能/parity/拓扑)⑦验收与测试策略(runbook)⑧关键决策留痕(ADR)。比例原则:小/机械 plan 可合并为一份短 spec、有意识扫一遍即可。
3. **任务级 checkbox 拆分**：每个任务/步骤用 `- [ ]` 显式拆分，粒度到"可单独勾选 / 验证"（bite-sized：写测试 / 跑红 / 实现 / 跑绿 / 提交 各一步）。
4. **每步 TDD**：先写测试（红）→ 最小实现（绿）→ 重构；**无测试不合并**。
5. **必含手动验收 runbook**：给出可照抄的命令序列 + 每步的期望证据，让人**不读代码也能复现验收**。**runbook 由 AI 在写计划时生成（不需人参与）；无 runbook 的计划不算完成。** 它"由人照着跑验收"发生在 §1.4，不在本阶段。
6. **争议设计落 ADR**：地基级 / 有争议的设计先决策、落 ADR，再进计划。
7. **全绿门禁**：每个任务结束跑 `<测试/lint/构建命令>` 全绿才进下一步；失败先系统化调试定位根因，不打补丁猜。
8. **实时更新状态（贯穿全程）**：每完成一步即勾 `- [x]` + 同步任务跟踪（in_progress / completed）；**不得做完一批才补勾或全程不更新**（无人值守跑完忘回填是典型反例）。

> 其中 **③拆分、④每步 TDD、⑤生成 runbook、⑦全绿门禁** 确定性高，可 headless 无人值守执行；需人参与的环节见 §1.3。

### 1.3 需要人参与的环节（不可全交给 AI）

以下关口**必须有人介入**，机器不得代替：

1. **探查结果研判**（对应 §1.2 探查优先）：探针跑完，由人确认事实文档可信、足以支撑写计划——不让 AI 拿不确信的"实测"就开写。
2. **DoR 过门研判**（对应 §1.2 DoR 就绪门）：由 owner / 独立评审者判 8 项是否真就绪（无静默 TBD、外部依赖已实测、对外契约已过第一个消费者）——机器自评不算就绪。
3. **争议设计拍板 → ADR**（对应 §1.2 争议设计落 ADR）：地基级 / 有争议的设计由**人**决策并落 ADR，不让 AI 自决。
4. **计划评审批准**（进入执行前）：计划成稿后人逐项对 spec/design 审——覆盖完整、拆分粒度合适、无占位符、runbook 可照抄跑；批准后才进入执行。
5. **跑 runbook 做验收**（§1.4 阶段，**非写计划阶段**）：runbook 本身由 AI 在写计划时生成（§1.2，不需人参与）；到 §1.4 必须由**人照着跑**并核对每步证据——机器自评不算。

### 1.4 隔离审查：机器自评 ≠ 验证

**核心命题**：测试是实现者自己写的，只覆盖他**想到的**路径，天然盲于他的盲区；机器审自己的代码复用同一套假设。因此——

> **计划"完成" ≠ 通过。** 合并前必须两道**独立于实现者**的审查，它们的价值正是去走实现者没设想到的路径：

- **① 人工验收**：照 runbook 手动验收 + 对照 spec 逐项核，用**真实、会撞边界**的数据（重复、残缺、越权、冲突……）。
- **② 独立 reviewer**：派**非实现者**的子代理 / 同事审本次 diff（`<base>..HEAD`），不带实现者的假设。

**Critical / Important 问题修完才可合并。headless / 自主执行尤其不得跳过——机器自评不算数。**

### 1.5 headless（无人值守）执行纪律

仅当**计划已审定、确定性高**时才用 headless；探索性 / 有未知判断的任务用交互式。

1. **订阅模式**：脚本先 `unset` API key 环境变量，避免误走计费；启动文件里硬编码的 key 须注释。
2. **流式 + 落日志**：执行输出流式打印并 `tee` 到日志文件；另开进程 tail 日志 / watch 提交记录看进度。
3. **合并前停**：headless 跑完只 push + 自评，**合并留给人**（§1.4 两道闸不可省）。

## 2. 工程纪律 **[硬]**

> 本章全部为硬纪律，覆盖决策、契约与通用工程实践。

### 2.1 决策与契约纪律

1. **ADR**：有争议 / 影响面大的决策一律落 ADR；正文为最新版，理由 / 历史进 ADR + git。
2. **先批判后批准**：地基级改动动手前做 blast-radius 分析，确认是真需求而非偏好——避免"中途重开已交付地基"的返工。
3. **契约优先（API-first）**：对外契约是稳定面；接口**先定契约（OpenAPI/proto）再实现**，契约入 git、CI 校验 breaking、client 由契约生成。
4. **薄适配，不裸传**：不直接暴露 / 透传底层系统，在其上加一层薄适配（投影 + 鉴权 + 隔离）。

### 2.2 通用工程纪律

1. **完成前必验证**：声称"完成 / 修复 / 通过"前**必须跑验证命令并给出证据**；证据先于断言。
2. **系统化调试**：遇 bug / 失败**先定位根因再改**，禁止猜测式打补丁。
3. **隔离开发**：功能在**独立分支 / worktree** 开发；不向 `<主分支>` 直接堆未评审改动。
4. **不静默砍范围**：任何降级 / 截断 / 抽样 / 跳过**必须显式记录**（日志 / PR 说明），不得让"覆盖了一部分"看起来像"全覆盖"。
5. **dev/prod parity**：本地开发环境含与生产同套依赖与功能,仅数据量/部署形态不同;不得用"本地缺依赖/用桩替代"作为 dev 默认(运行时依赖归运行时、测试桩仅限 CI)。

### 2.3 CI 防线（机器强制）

人会忘、会赶，所以把能自动化的纪律交给 CI 守：

- `<测试命令>` 全绿（单元 + 集成）。
- `<lint / 类型检查 / 格式化命令>` 通过。
- 契约 breaking-change 检查 / 代码生成产物新鲜度检查。
- `<其他项目特定门禁：依赖审计 / 覆盖率阈值 / 构建可复现……>`

### 2.4 反模式（禁止）

| 反模式 | 正解 |
|---|---|
| spec 没就绪就开写 tasks | 先过 DoR 八项就绪门（§1.2 DoR）|
| 详细设计前置塞进 sprint 文档 | sprint 给摘要,详细 just-in-time 进每个 plan（§1.1）|
| headless 跑完 = 完成 | 必过人工 runbook + 独立 reviewer（§1.4） |
| 机器审自己的代码当审查 | 隔离审查（独立子代理 / 人）（§1.4） |
| 把猜的外部 API 写进计划 | 探针优先，实测钉死（§1.2 探查优先） |
| 对外契约不经消费者就冻结 | 冻结前过"第一个消费者":UI 低保真原型 / 程序 SDK·样例（§1.2 DoR ②）|
| 测试用随机唯一值 / 假对象，永不撞边界 | 加重复 / 缺字段 / 越权 / 冲突等用例 |
| 地基改动直接动手 | 先批判（blast radius）+ ADR（§2.1 先批判后批准） |
| 计划做完一批才补勾 | 实时勾选 + 任务跟踪（§1.2 实时更新状态） |
| 裸传底层系统的 API | 自有契约 + 薄适配层（§2.1 薄适配） |
| 静默降级 / 截断 / 跳过 | 显式记录（§2.2 不静默砍范围） |
| 本地缺依赖用桩当 dev 默认 | dev/prod parity（§2.2）|
